# Session 3 — 2026-03-12

## Experiment

**Halve batch size further: 2^17 → 2^16 with DEVICE*****BATCH*****SIZE=32**

Rationale: Sessions 1-2 showed consistent improvements from smaller batch sizes (more optimizer steps). Tested whether the trend extends to 2^16.

## Changes

- `TOTALBATCHSIZE`: 2^17 → 2^16 (~65K tokens per step)
- `DEVICEBATCHSIZE`: 64 → 32

## Analysis

- The smaller-batch trend has hit diminishing returns
- 2557 steps (up from 1357) but val_bpb got worse (1.0584 vs 1.0472)
- Training loss was noisier and higher toward end (~2.97 vs ~2.94)
- Gradient noise from batch size 32 is too high for effective optimization
- **2^17 appears to be the sweet spot for batch size** with this model configuration
- Total tokens processed was similar (167.6M vs 177.9M), confirming this is a noise issue

## Conclusion

The batch size → optimizer steps tradeoff has a sweet spot around 2^17 for this model. Further improvements need to come from other dimensions: learning rates, architecture, or schedule tuning.

- Try higher matrix LR (0.06 → 0.08)

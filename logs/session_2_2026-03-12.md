# Session 2 — 2026-03-12

## Experiment

**Halve batch size again: 2^18 → 2^17 with DEVICE*****BATCH*****SIZE=64**

Rationale: Session 1 showed that reducing batch size from 2^19 → 2^18 doubled optimizer steps (350 → 686) and improved val*bpb from 1.0776 to 1.0591. Continuing this trend by halving again to 2^17 with DEVICE*BATCH_SIZE=64.

## Changes

- `TOTALBATCHSIZE`: 2^18 → 2^17 (~131K tokens per step)
- `DEVICEBATCHSIZE`: 128 → 64

## Results

| Metric | Value |

|--------|-------|

| val_bpb | **1.0472** ✓ (prev best: 1.0591) |

| Peak VRAM | 22.7 GB (down from 43.8 GB) |

| Steps | 1357 (up from 686) |

| MFU | 14.21% (down from ~28%) |

| Training seconds | 300.2 |

| Total tokens | 177.9M |

| Params | 50.3M |

**Status: KEEP** — new best val_bpb.

## Analysis

- The trend holds: more optimizer steps in the fixed time budget = better val_bpb
- ~2x more steps (686 → 1357) gave another −0.012 improvement
- VRAM dropped dramatically (43.8 → 22.7 GB) since we're processing half the tokens per forward pass
- MFU dropped from ~28% to 14.2% — the GPU is less utilized per step, but total tokens processed is similar (~178M vs ~180M)
- Total tokens is roughly the same, confirming the gain comes from more frequent parameter updates, not more data
- We're now at epoch 1, meaning we've cycled through the training data at least once
